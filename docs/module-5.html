Module 5 — Review
5

Review & prevention

~30 min

Learning objectives

  • Run /code-review as a two-axis review — Standards ‖ Spec — without merging findings across axes.
  • Close every production bug with Root Cause + Prevention on the ticket.
  • Know when ticket notes are enough vs when a thin Incident + Postmortem is required (outage-class severity/s1).
  • Name the process metrics the team tracks and the improvement cadence.

Hard truth

A clean diff that fixes the wrong symptom ships confidently wrong behavior. Spec without Standards leaves landmines. Closing without Prevention guarantees the same bug class returns.

Closure workflow

  1. Open a PR that references the ticket and states the winning hypothesis.
  2. Run /code-review (Standards ‖ Spec) against a pinned fixed point.
  3. Address feedback on both axes.
  4. Update the ticket with Root Cause + Prevention (template below).
  5. QA verifies the fix (Fixed → Verified) before true close.
  6. Close ticket and PR. File architecture / monitoring / docs follow-ups as separate tickets when needed.

/code-review — Standards ‖ Spec

Mandatory for production bug PRs. Run two axes in parallel — do not merge or re-rank findings across axes:

AxisQuestion
StandardsDoes the diff follow repo coding standards and smell baseline?
SpecDoes the diff faithfully fix the originating issue (right behavior)?

Process essentials

  1. Pin a fixed point (main, merge-base, or SHA). Confirm a non-empty three-dot diff.
  2. Identify the spec source (issue refs, PR description, acceptance criteria).
  3. Identify standards sources (CODING_STANDARDS.md, CONTRIBUTING.md, repo conventions).
  4. Spawn parallel Standards and Spec reviews.
  5. Aggregate under separate headings; one-line summary per axis — never re-rank across axes.

Why both axes matter for bugs

  • Clean code that fixes the wrong symptom → Standards pass, Spec fail.
  • Exact fix that ignores project conventions → Spec pass, Standards fail.

Root Cause + Prevention (mandatory)

Every closed production bug includes both fields on the ticket. Prefer causal mechanism over labels like “null pointer.”

## Closure

**Root Cause:**
[Mechanism — what broke and why.]

**Prevention:**
- [ ] Regression test at seam: …
- [ ] Guard / validation: …
- [ ] Docs / CONTEXT.md term clarified: …
- [ ] Follow-up architecture ticket (if seam missing): …

Example Root Cause: “Timezone offset from Safari was sent as undefined after refresh; server rejected the header; UI treated the error body as an empty invoice list.”

Incident + Postmortem (thin rules)

Most bugs follow report → triage → diagnosis → fix → review → close only. Escalate to Incident handling when:

  • Customer-visible outage or data loss
  • Multi-team response required
  • Concentrated investigation stuck beyond a short window without a restore path
  • On-call / monitoring failure that hid the issue

Incident (thin)

  • Declare explicitly; restore service first, deep RCA second.
  • Keep a live state note (what’s known, what’s tried, next action, owner).
  • Do not fold full ICS / war-room role theatre into routine Diagnosis Loop work.

Postmortem

  • Required for outage-class severity/s1 Incidents (user-visible downtime, data loss, on-call intervention, long time-to-mitigate, monitoring miss).
  • Blameless write-up with owned preventive actions and a review pass — above-and-beyond ticket Root Cause + Prevention.
  • Routine bugs: ticket RCA + Prevention is enough. Not every severity/s1 label needs a Postmortem if it was not an Incident.

Paper Exercise E — Two-axis review

Fictional PR description: “Fixed empty invoices by removing Accept-Timezone support globally.”

List at least:

  • Spec findings (vs the bug: the header should be honored, not removed).
  • Standards findings (speculative generality / shotgun surgery if many call sites changed).

Pass criteria: at least one Spec fail on “remove header globally.”

Live practice

  1. Run /code-review against a training or Exercise Lab PR; classify each finding as Standards or Spec.
  2. Write a one-sentence Prevention note for a past painful bug, then expand to three concrete actions (test, guard, docs/follow-up).

Checklist

Self-check

A PR removes Accept-Timezone support entirely to “fix” empty invoices. Which axis fails first, and why?

Spec fails: the originating bug requires honoring the header, not deleting the feature. Standards may also fail if the change is shotgun surgery across many call sites — list those separately; do not merge axes.

When is a Postmortem required beyond ticket Root Cause + Prevention?

For outage-class severity/s1 Incidents (user-visible downtime, data loss, on-call intervention, long TTM, monitoring miss) — not for every routine bug that happens to carry an S1 label.
