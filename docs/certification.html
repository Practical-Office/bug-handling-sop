Bug Handling certification

Not checkbox-only. Assemble linked proof, self-attest, then get a lead review.

What “certified” means

You can run the Five-Module Course end-to-end: report → triage → diagnosis (red Phase 1) → fix → Standards ‖ Spec review → Root Cause + Prevention — and you can show evidence, not just claim it.

Evidence Pack requirements

Submit one pack (gist, PR comment, Notion page, or ticket comment thread) that includes all of the following. Self-attest checkboxes below; a lead must still review the pack.

  1. Rewritten report OR Paper Exercise A
    • Full mandatory template (Title, Environment, Reproduction Steps, Expected, Actual, Logs/Screenshots, Business/User Impact).
    • Either rewrite a real weak report from your work, or complete Exercise A from Module 1 (“Dashboard is broken on mobile sometimes. Customers are complaining.”) with assumptions marked ASSUMPTION:.
    • Link or paste the deliverable in the pack.
  2. Phase 1 command + output
    • One red-capable command you have already run that asserts the reporter’s symptom.
    • Source: Exercise Lab (exercises/invoice-api Phase 1 script) or a real bug you diagnosed with /diagnosing-bugs.
    • Include the command line and enough output to show the failure (red signal). No BookIQ product backlog required for course cert.
  3. Root Cause + Prevention
    • From a closed ticket or from the Exercise Lab fix path.
    • Use the Living SOP closure template: mechanism (not a vague label) + concrete Prevention actions (regression test, guard, docs, architecture follow-up as applicable).
  4. Lead review
    • A team lead (or designated reviewer) confirms the pack meets the three evidence items and that you completed Setup + Modules 1–5 checklists.
    • Lead notes any gaps; pack is not “done” until gaps are closed or explicitly waived in writing.
