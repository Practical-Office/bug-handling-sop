Why a shared bug process matters

Hub intro · not a numbered module

Without a common loop, every engineer invents intake, triage, and fix habits. Incomplete tickets, mixed severity language, and untested patches burn time. This course teaches one end-to-end path so the team ships fewer regressions and closes with Root Cause + Prevention.

Principles

Reproducibility First (fulfilled by Diagnosis Loop Phases 1–2) · Skills-driven · Root Cause + Prevention · /tdd + /code-review mandatory · Improve quarterly

End-to-end flow

Report → (/grill-*) → /triage → /diagnosing-bugs → /tdd → PR → /code-review → QA → RCA/Prevention → Close
                                                                              → (outage S1) Incident + Postmortem

Report — Bug Reporting Standards

~25 min

Hard truth

Incomplete tickets get returned. Reporting is the first feedback loop — do not guess missing fields.

Mandatory template

**Title:** [Clear, concise description]

**Environment**
- Browser / OS / Version:
- Backend / API version:
- User role / permissions:
- Other relevant context:

**Reproduction Steps**
1.
2.
3.

**Expected Behavior**
**Actual Behavior**
**Logs / Screenshots / Video**
**Business / User Impact**

Tips

  • STR ≤ ~6 steps; one clear failure mode.
  • Impact specific (who / how many / workaround?).
  • Vague intake → /grill-me or /grill-with-docs.

2

Triage Process

~30 min

Hard truth

Triage is confirmation + routing. Work starts only at ready-for-agent or ready-for-human.

Severity ≠ Priority

Severity = impact (how bad). Priority = schedule (when). Do not collapse into a mixed P1–P4 impact scale.

SeverityMeaningTriage SLA
severity/s1Critical / no acceptable workaround≤ 1h
severity/s2Major / complex or partial workaround≤ 4h
severity/s3Moderate / reasonable workaroundNormal cadence
severity/s4Low / cosmeticNormal cadence

Priority: priority/p0 now · p1 next · p2 soon · p3 when capacity.

Triage Category + State

Category (one): bug | enhancement.

State (one): needs-triage · needs-info (~14d stale) · ready-for-agent · ready-for-human · wontfix.

Paper Exercise B: calibrate S/P + state on module-2.html.

3

Diagnosis Mastery

~45 min

Hard truth

Diagnosis without a minimized reproduction is guessing. Build a red-capable feedback loop before hypothesising.

Diagnosis Loop — six phases (/diagnosing-bugs)

  1. Build a feedback loop — one red-capable, deterministic, agent-runnable command you have already run.
  2. Reproduce + minimise — confirm the user’s failure; shrink until every element is load-bearing.
  3. Hypothesise — 3–5 ranked, falsifiable predictions before testing any.
  4. Instrument — one variable at a time; tagged probes.
  5. Fix + regression test — preview here; Module 4 owns the green path via /tdd.
  6. Cleanup + post-mortem — remove debug noise; winning hypothesis; prevention question.

Reproducibility First is a principle fulfilled by Phases 1–2 — not a competing top-level “reproduce then patch” shortcut.

4

Fixing with /tdd

~35 min

Hard truth

TDD without a failing test first is theater. Speculative refactors in a bug PR create new regressions.

Prerequisites

  • Red Phase 1 from Module 3 / lab.
  • Minimisation story + ranked hypotheses.
  • Agreed correct seam for the regression test.
  • On real tickets: Triage State ready-for-*.

Rules

  1. Red before green — failing test first, then minimal fix.
  2. One vertical slice at a time at a confirmed seam.
  3. Refactor is not part of red→green — park for /code-review (Module 5).
  4. Re-run the original un-minimised Phase 1 command after green.

Continue Invoice API lab on module-4.html — not product backlog.

5

Review & prevention

~30 min

Closure workflow

  1. PR references ticket + winning hypothesis.
  2. /code-review — Standards ‖ Spec (parallel; never merge axes).
  3. Ticket: Root Cause + Prevention.
  4. QA verifies → close. Follow-ups as separate tickets.

Standards ‖ Spec

  • Standards: Does the diff follow repo conventions?
  • Spec: Does the diff faithfully fix the originating issue?

Closure template

**Root Cause:** [mechanism — what broke and why]
**Prevention:**
- [ ] Regression test at seam: …
- [ ] Guard / validation: …
- [ ] Docs / CONTEXT.md: …
- [ ] Architecture follow-up (if seam missing): …

Incident + Postmortem (thin)

Escalate when customer-visible outage, multi-team response, or stuck investigation. Restore first. Blameless Postmortem required for outage-class severity/s1 Incidents — not every S1 label.

Metrics

Track by severity/s* (not mixed P1–P4 as impact). Triage SLA hit rate · % with red Phase 1 · % with regression tests · recurring rate · needs-info age · team feedback.

Certify with an Evidence Pack. Live teaching: module-5.html.
