/tdd

Test-driven development

Build-phase skill · invoked by /implement

The red → green loop. This skill is the reference that makes the loop produce tests worth keeping: what a good test is, where tests go, anti-patterns, and the rules of the cycle.

← Module 4: Build skills /code-review /implement

Why

Build-phase code in an experiment should be correct and changeable. TDD forces you to name behavior through public interfaces before implementation hardens. Tests that survive refactors become living specification — especially valuable when the experiment Build is thin but the learning stakes are high.

Every red → green cycle consults this skill before and during the loop, not after. Refactoring belongs to /code-review, not to the implementation loop itself.

When

  • /implement reaches code that benefits from automated verification at agreed seams
  • The spec or ticket names testable behavior (not “refactor internals”)
  • You and the agent have confirmed which seams are under test
  • User explicitly asks for test-first work or mentions red-green-refactor

Skip or minimize for throwaway /prototype code, one-off scripts, or Builds where the spec explicitly defers tests (document why on the issue).

How

  1. Confirm seams. Before any test, write down the public boundaries under test and agree them. Ask: “What’s the public interface, and which seams should we test?”
  2. Red. Write one failing test at one seam. Expected values come from an independent source of truth — spec literal, worked example, known-good fixture — not from re-deriving what the code does.
  3. Green. Write only enough code to pass that test. No speculative features for future tests.
  4. Repeat. One vertical slice: one seam, one test, one minimal implementation per cycle.
  5. Hand off. When the ticket’s behavior is covered, /implement runs typecheck, targeted tests, full suite once, then /code-review.

Read project CONTEXT.md (if present) so test names match domain vocabulary. Respect ADRs in the area you touch.

Inputs / outputs

InputsOutputs
Agreed seams (from spec or /to-spec) Failing test at one seam (red)
One behavior slice from the ticket Minimal implementation passing that test (green)
Independent expected values (spec, fixture, literal) Tests that read like specification — “user can complete X when Y”
Project test conventions + domain language Committed code on the experiment branch, ready for review

Anti-patterns

  • Implementation-coupled tests — mock internal collaborators, test private methods, or assert via side channels (e.g. querying the database instead of the interface). Tell: test breaks on refactor though behavior unchanged.
  • Tautological tests — assertion recomputes the expected value the same way the code does (expect(add(a,b)).toBe(a+b)). Can never disagree with the implementation.
  • Horizontal slicing — all tests first, then all implementation. Bulk tests verify imagined behavior; structure commits before you understand the code. Use vertical slices instead.
  • Refactor inside the loop — cleaning structure during red-green. Defer to /code-review.
  • Unconfirmed seams — writing tests at boundaries nobody agreed to. Effort lands on noise instead of critical paths.

Micro-scenario

An experiment Build adds a “mark insight reviewed” action. The ticket says: when a user marks reviewed, the insight status becomes reviewed and a timestamp is recorded.

Seams agreed: public API on the insight service — markReviewed(insightId) — not the database layer.

Cycle 1: Red — test calls markReviewed on a pending insight; expect status reviewed. Green — minimal handler update.

Cycle 2: Red — test expects reviewedAt set. Green — add timestamp. Stop — no extra audit log until a ticket asks for it.

Review stage catches Feature Envy or duplication; TDD stage only proved the behavior.
