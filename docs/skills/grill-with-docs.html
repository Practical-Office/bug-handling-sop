Grill with docs

A relentless interview to sharpen a plan or design — and capture glossary entries and architecture decision records as decisions crystallize.

Why

Shared understanding before specs and code

Vague experiments waste Build time. grill-with-docs combines the discipline of grilling (one question at a time until you agree) with domain modeling: terms get pinned in CONTEXT.md, hard-to-reverse choices become ADRs, and contradictions with existing code surface early.

You do not ship from this skill. You stop when the human confirms shared understanding — then to-spec or implement can proceed without re-interviewing.

When

Use this skill

  • Moving an experiment from Backlog to Build with non-obvious scope
  • Before to-spec on any non-trivial build
  • At Learn when pivoting and the domain language may shift
  • When the issue template’s technical context needs sharpening (@paths hints)

Skip for tiny builds (single small PR) — optional light grill only.

How

Steps

  1. Start a grilling session — invoke grilling: one question at a time, wait for each answer, offer a recommended answer with every question.
  2. Look up facts — explore the repo, issue body, and metrics docs yourself; reserve the human for decisions only.
  3. Apply domain modeling — challenge glossary conflicts, sharpen overloaded terms, stress-test edge-case scenarios, cross-check claims against code.
  4. Write docs inline — update CONTEXT.md when a term resolves; offer ADRs only when a choice is hard to reverse, surprising without context, and the result of a real trade-off.
  5. Confirm stop — do not act on the plan until the human confirms shared understanding.

Inputs / outputs

InputsOutputs
Experiment issue (hypothesis, build scope, measure table); relevant repo paths; existing CONTEXT.md / ADRs Aligned mental model; updated glossary; new or revised ADRs; explicit open questions resolved — ready for to-spec

Anti-patterns

What to avoid

  • Batch questions — Multiple questions at once bewilders; grilling is strictly one-at-a-time.
  • Asking what tools can answer — File layout, label lists, and template fields should be looked up, not interrogated.
  • Implementing before confirm — No code or spec publication until shared understanding is explicit.
  • ADR spam — Not every choice deserves an ADR; trivial or easily reversed decisions stay in conversation only.
  • CONTEXT as a spec — Glossary only; implementation detail belongs in specs and tickets.

Micro-scenario

Abstract example

An engineer opens experiment issue #22 (“customers abandon multi-step forms”). They run /grill-with-docs with @paths to the onboarding module.

The agent asks: “Does ‘abandon’ mean closed tab, or saved draft never submitted?” — recommends counting only submitted-or-explicit-dismiss events. The human agrees; CONTEXT.md gains an abandonment entry.

A second question probes kill criteria vs. sample size. The agent reads the measure table from the issue rather than asking for numbers already there.

When the human says partial save is in scope, the agent finds code that only tracks full submits, flags the mismatch, and offers an ADR on event schema — declined for now as reversible.

Session ends with explicit “we’re aligned.” /to-spec is the next step.
