Wayfinder

Planning skill · decision maps on your issue tracker

Plan a huge chunk of work — more than one agent session can hold — as a shared map of decision tickets. Resolve them one at a time until the route to the destination is clear. Wayfinding finds the way; it does not charge at the destination.

← Module 5: Running an experiment /prototype /grilling

Why

Some efforts arrive wrapped in fog: the destination is visible, but the sequence of decisions to get there is not. A single experiment ticket is the right unit for one hypothesis. A multi-quarter platform bet, pricing overhaul, or new product surface needs a map — an index of decisions, not a Gantt of deliverables.

Wayfinder keeps planning honest: each ticket resolves a question, default mode is plan-not-do, and the map is done when nothing remains to decide before someone executes. The pull to “just build it” usually means you’ve reached the edge of the map and should hand off to specs, experiment tickets, or implementation.

This course is not a wayfinder map. These modules teach the Bug Handling SOP system. Use wayfinder on your product’s fuzzy efforts — not to re-chart this onboarding course.

When

  • A loose idea is too big for one session and the path is unclear
  • You need a shared index of open decisions collaborators can claim independently
  • Research, prototypes, or grilling must happen in dependency order before committing to Build
  • You’re deciding what to experiment on or how to structure a bet — not running a single experiment through Measure/Learn

Not for tracer-bullet implementation tickets (use /to-tickets), single experiment execution (use Module 5), or charting training content you’re already taking module-by-module.

How

Chart the map (one session)

  1. Name the destination. Grilling + domain modeling — spec to hand off, decision to lock, or change to make in place.
  2. Map the frontier. Breadth-first: surface open decisions and first takeable steps. If the way is already clear and small, you may not need a map — ask how to proceed.
  3. Create the map issue (label wayfinder:map): Destination, Notes, empty Decisions so far, fog in Not yet specified, Out of scope.
  4. Create child tickets you can specify now; wire blocking in a second pass. Unspecifiable fog stays in Not yet specified.
  5. Fire research subagents for research tickets in parallel. Stop — charting resolves nothing.

Work through the map (one ticket per session)

  1. Load the map — low-res index, not every ticket body.
  2. Choose a frontier ticket (open, unblocked, unclaimed). Claim by assigning before any work.
  3. Resolve via the ticket type: research (AFK), prototype (HITL), grilling (HITL), task (unblocks a decision).
  4. Record resolution as a comment, close the ticket, append gist + link to map Decisions so far.
  5. Graduate fog into new tickets; rule mis-scoped work out of scope. Never resolve more than one ticket per session (except parallel research during charting).

Refer by name

In human-readable narration, use ticket titles, not bare ids. Names wrap links; #42 walls are illegible.

Ticket types

TypeModeResolves via
researchAFK/research subagent — external docs, APIs, facts
prototypeHITL/prototype — concrete artifact to react to
grillingHITL/grilling + domain modeling — default case
taskHITL or AFKManual work that unblocks a decision (access, data move) — not the destination itself

Anti-patterns

  • Execution map. Tickets that are build slices instead of decisions — use /to-tickets after the way is clear.
  • Pre-slicing fog. Ticketizing what you cannot yet phrase sharply; one fog patch may become several tickets or none.
  • Multi-resolve sessions. Closing three tickets in one sitting — loses reviewability and concurrent claim safety.
  • Agent stands in for HITL. A grilling ticket where the agent answers its own questions.
  • Map as store. Pasting full decisions into the map body instead of gists + links on child tickets.
  • Re-charting this course. Wayfinder on Bug Handling SOP itself — you’re already in the guided module path.

Micro-scenario

Destination: “Experiment program for a new analytics surface — know which three hypotheses to run in Q1.”

Map created. Frontier tickets: “Which behaviors are falsifiable in week one?” (grilling), “What do competitors expose in their APIs?” (research, blocks instrumentation decision), “Task: sample export from staging” (task, blocks data-shape grilling).

Research closes — API lacks act-level events; grilling revises hypotheses to view-based proxies. Fog graduates: “Do we need a prototype insight card before ticket drafting?” → new prototype ticket.

When the last open ticket closes, the map hands off: three experiment issues filed on the Bug Handling SOP board (Backlog), each with hypothesis + kill criteria. Wayfinder stops; Module 5 execution begins.
