/review

Code review

Build-phase skill · invoked by /implement

Review the diff since a fixed point along two axes — Standards (repo conventions + smell baseline) and Spec (does the change match what was asked?). Report them side by side; never merge the axes.

← Module 4: Build skills /tdd /implement

Why

A change can pass one axis and fail the other. Code that follows every standard but implements the wrong thing is still wrong. Code that matches the spec but violates project conventions creates drag for the next experiment Build.

Separating Standards and Spec stops one axis from masking the other. In Bug Handling SOP work, the spec source is usually the experiment issue, a published spec from /to-spec, or a tracer-bullet ticket from /to-tickets.

When

  • After /tdd cycles complete for a ticket — /implement invokes review before commit
  • User asks to review a branch, PR, or WIP changes since a commit, tag, or branch
  • Before merging experiment Build work that affects what Measure will observe

Not for Learn-phase decisions, metric interpretation, or hypothesis validation — those are human-owned (see Module 5).

How

  1. Pin the fixed point. Commit SHA, branch, tag, main, or HEAD~N. Diff: git diff <fixed-point>...HEAD (three-dot). Confirm non-empty before proceeding.
  2. Identify the spec source. Issue refs in commits → fetch from tracker; else user path; else spec under docs/ or specs/. If none, Spec axis reports “no spec available.”
  3. Identify standards sources. CODING_STANDARDS.md, CONTRIBUTING.md, etc. Always include the Fowler smell baseline (repo standards override smells).
  4. Run both axes in parallel. Standards sub-agent: violations + smells per hunk. Spec sub-agent: missing requirements, scope creep, wrong implementations — quote spec lines.
  5. Aggregate. Present under ## Standards and ## Spec headings. One-line summary: count per axis, worst issue per axis. Do not pick a single winner across axes.
  6. Fix and commit. Address findings; /implement commits when both axes are acceptable for the ticket scope.

Inputs / outputs

InputsOutputs
Fixed point ref + git diff …HEAD Standards report: documented violations + labelled smells
Originating issue, PRD, or ticket body Spec report: gaps, creep, suspicious implementations
Repo standards docs + smell baseline Side-by-side findings (not merged or reranked)
Commit list git log fixed..HEAD --oneline Summary line: totals + worst issue per axis

Standards smell baseline (judgement calls)

Always apply unless repo docs endorse the pattern. Each smell is a heuristic, not a hard violation.

  • Mysterious Name — rename; murky design if no honest name exists
  • Duplicated Code — extract shared shape
  • Feature Envy — move method to the data it uses
  • Data Clumps / Primitive Obsession — introduce a small type
  • Repeated Switches — polymorphism or shared map
  • Shotgun Surgery / Divergent Change — gather or split modules by change reason
  • Speculative Generality — delete unused abstraction
  • Message Chains / Middle Man — hide navigation or remove delegate layer
  • Refused Bequest — prefer composition over bad inheritance

Anti-patterns

  • Single-axis review. “Looks clean” while implementing the wrong metric event ID.
  • Spec axis without a spec. Guessing intent from code shape instead of fetching the issue.
  • Merging findings. Ranking Standards issues as more important than Spec misses wrong behavior.
  • Review as Learn. Using review to decide if the experiment succeeded — that needs Measure numbers.
  • Refactor theater. Large structural changes during experiment Build that the ticket did not require.

Micro-scenario

Ticket: “Emit insight.acted when user completes mark-reviewed.” Fixed point: main.

Standards axis: flags Feature Envy — handler reaches into three repository fields instead of calling insight.markReviewed().

Spec axis: passes event name and trigger; flags missing tenant_id prop required by the metrics dictionary.

Fix both before commit. Neither axis alone caught the full gap. Learn column still waits for four weeks of act-rate data.
