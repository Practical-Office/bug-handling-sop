Grilling

Interview relentlessly about every aspect of a plan until you reach shared understanding — one question at a time.

Why

Stress-test before commitment

Plans hide assumptions. Grilling walks the decision tree branch by branch, resolving dependencies between decisions in order. Each question forces a explicit choice; the agent supplies a recommended answer so you are never staring at a blank prompt.

grill-with-docs wraps this interview with domain modeling and doc capture. Use raw grilling when you need the interview discipline without ADR/glossary writes.

When

Use this skill

  • User says “grill this,” “stress-test,” or wants decisions challenged
  • Inside grill-with-docs as the interview engine
  • Before committing to a pivot at Learn
  • Any time multiple dependent decisions must be resolved sequentially

How

Steps

  1. Frame the subject — plan, decision, or idea under test.
  2. Walk the tree — pick the next unresolved dependency; ask exactly one question.
  3. Recommend — with each question, state your recommended answer and why.
  4. Wait — do not ask the next question until the human responds.
  5. Look up facts — if the answer exists in the environment (files, issues, APIs), fetch it instead of asking.
  6. Confirm understanding — when the tree is resolved, summarize and wait for explicit confirmation before any action.
  7. Do not act — no implementation, spec, or tickets until confirmation.

Inputs / outputs

InputsOutputs
Plan, decision, or design under review; access to repo/issue context for factual lookups Resolved decision tree; explicit shared understanding; human confirmation — no artifacts unless paired with grill-with-docs

Anti-patterns

What to avoid

  • Question dumps — Listing five questions at once defeats the purpose.
  • Interrogating facts — “What’s in file X?” is an agent job, not a grill question.
  • Premature action — Shipping code or filing specs before “we’re aligned.”
  • Rhetorical grilling — Questions without recommended answers leave the human doing all the work.
  • Skipping confirmation — Assuming silence means agreement.

Micro-scenario

Abstract example

A lead proposes measuring “engagement” via weekly active users for a new checklist feature.

Q1: “Should engagement mean completed checklist or merely opened?” — recommends completed because the hypothesis is about setup completion. Human picks completed.

Q2: “Is the population all signups or only self-serve?” — agent reads the issue sample field instead of asking. Finds “self-serve only” already stated.

Q3: “If completion rises but support tickets also rise, is that pass or fail?” — recommends fail unless a secondary metric is added. Human adds ticket volume to kill criteria.

Agent summarizes the three decisions; human confirms. Session ends — no spec written yet.
