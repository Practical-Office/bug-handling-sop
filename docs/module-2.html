Triage Process

Learning objectives

What you will leave able to do

  • Apply Severity (severity/s1–severity/s4) and Priority (priority/p0–priority/p3) with correct triage SLAs.
  • Route issues with Matt’s Triage Category + Triage State machine (/triage).
  • Verify claims (reproduce or needs-info) before declaring work ready — including stale and non-bug routing.

Hard truth

Triage is confirmation + routing, not “add a label and hope.” Peer SOPs (Kubernetes, Chromium, GitLab, Mozilla) agree: reproduce or clarify first, then accept into a work queue. Work starts only at ready-for-agent or ready-for-human.

Living SOP §5

Severity ≠ Priority

Severity = user/business impact (how bad). Priority = scheduling urgency (when we plan to work it), given Severity and capacity. Do not collapse these into a single mixed P1–P4 impact scale. Canonical label strings: triage-labels.md.

Severity (severity/s1 … severity/s4) — workaround-aware

LabelMeaningWorkaround lensTriage SLA
severity/s1 Critical: outage, data loss/corruption, active security risk, or core flow blocked for many users No acceptable workaround (or workaround is unsafe) ≤ 1 hour
severity/s2 Major: significant breakage for many users, or severe degradation of a primary workflow Workaround exists but is complex, costly, or partial ≤ 4 hours
severity/s3 Moderate: limited users, secondary paths, or non-critical features Reasonable workaround available Normal cadence (daily skim; full triage within the working week)
severity/s4 Low: cosmetic, typos, polish, minor UX friction Trivial workaround or negligible impact Normal cadence

Calibration: when unsure between two severities, overestimate briefly, then confirm with a domain expert — do not leave hot bugs under-labeled.

Priority (priority/p0 … priority/p3)

LabelMeaning
priority/p0Schedule now — drop / interrupt current work
priority/p1Next — current sprint / immediate queue
priority/p2Soon — planned backlog (near term)
priority/p3When capacity allows

Typical pairing (guidance, not automatic): S1 → usually P0/P1; S2 → often P1/P2; S3 → often P2/P3; S4 → usually P3. A high-Severity bug can wait if mitigated; a lower-Severity bug can be priority/p0 when it blocks a release or customer commitment.

AI-generated triage comments must start with:

> *This was generated by AI during triage.*

Triage steps (Living SOP §6.3)

  1. Ensure the report template is complete enough to evaluate (else needs-info).
  2. Apply exactly one Triage Category and exactly one Triage State.
  3. Apply Severity and Priority.
  4. Add component/area labels (e.g. auth, dashboard, mobile).
  5. Verify the claim — for bugs: reproduce from STR or agree the STR is sufficient. Outcomes: confirmed / failed / insufficient detail.
  6. Route: assign owner, place on backlog, or attach agent/human brief for ready-for-*.
  7. Deduplicate; close duplicates as wontfix with a pointer to the canonical ticket.

Needs-info stale policy (~14 days)

  1. Request missing information once using the needs-info template.
  2. Nudge if there is no reply after ~7 days.
  3. If ~14 days pass with no reporter reply, close (or re-close) with a polite note that the ticket can be reopened when details arrive.
  4. When the reporter replies, move needs-info → needs-triage and re-evaluate.

Non-bug routing

Keep the bug queue for defects. Route other intake quickly:

SignalAction
Duplicatewontfix + link to canonical issue
Support / how-to / questionClose or redirect to support channel; not bug work
Feature requestCategory enhancement (not severity theater on a fake bug)
Already implementedwontfix + pointer to code/docs
Out of scope / rejectedwontfix + record per Matt triage skill

Security — private advisory path

Do not file public issues with exploit details, secrets, or reproducible attack steps.

  • Prefer the repository’s private vulnerability reporting / security advisory flow (GitHub private reporting and/or SECURITY.md where present).
  • Triage may still assign high Severity once the report is in the private channel.
  • Public tickets, if any, stay high-level (“security issue under private advisory”) without payload.

Paper Exercise B

Calibrate Severity, Priority, and state

For each scenario, assign severity/s*, priority/p*, and a Triage State (plus Category if not obviously a bug). Then reveal the facilitator key.

#Scenario
1Login 500s for all users in production; no workaround
2CSV export wrong column order for one enterprise tenant; they can export via API as a temporary workaround
3Typo on settings empty state
4XSS in a shared comment field (active risk)
5“App feels slow sometimes” with no metrics, STR, or environment
  1. Login 500s: severity/s1, usually priority/p0; needs-triage → verify → ready-for-human or ready-for-agent with brief. Treat as Incident if customer-visible outage needs multi-team restore (Module 5).
  2. CSV column order (one tenant, API workaround): often severity/s3 (reasonable workaround) or severity/s2 if the tenant’s primary workflow is blocked and workaround is costly; Priority often priority/p1–priority/p2. May start needs-info for a sample file / expected columns.
  3. Typo: severity/s4, priority/p3; can go ready-for-agent quickly once confirmed.
  4. XSS: impact often severity/s1 (or severity/s2); use the private security path — not public exploit writeups. Priority usually priority/p0/priority/p1.
  5. “Feels slow sometimes”: do not severity-guess — needs-info first (metrics, STR, environment). Close after ~14d with no reply per stale policy.

Checklist

Self-check

What is the triage SLA for severity/s1? Is that the same as Priority?

First triage response within 1 hour. That is a Severity-driven SLA — Priority is a separate scheduling label (often priority/p0 or priority/p1 for S1, but not automatic).

When does diagnosis / fix work start?

Only when Triage State is ready-for-agent or ready-for-human — after the claim is verified (or STR agreed sufficient), not while stuck in needs-triage or needs-info.
