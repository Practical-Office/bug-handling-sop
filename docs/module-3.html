Diagnosis Mastery

~45 min

Learning objectives

  • Run /diagnosing-bugs through all six phases — skip only when explicitly justified.
  • Build a tight, red-capable, agent-runnable feedback loop before hypothesising.
  • Produce 3–5 falsifiable, ranked hypotheses and instrument one variable at a time.
  • Use the in-repo Exercise Lab — not a product backlog ticket.

Hard truth

Diagnosis without a minimized reproduction is guessing. Jumping to a theory before a red-capable command exists is the failure mode this skill exists to prevent.

Skill & principle

  • Mandatory skill: /diagnosing-bugs — the Diagnosis Loop (six phases; feedback loop first).
  • Reproducibility First is a principle, satisfied by Phases 1–2. It is not a competing “reproduce then patch” shortcut that replaces Matt’s order.
  • Hand-off: stop before the production fix — Module 4 owns /tdd and the green path. Phase 5 here means knowing where the regression test and fix land.

Read CONTEXT.md and relevant ADRs before deep exploration on a real codebase. For this module, the lab README is enough.

Common failure modes

  • Hypothesis before loop: reading code to “get a theory” before any red command exists.
  • Wrong bug: a loop that fails for a nearby error, not the reporter’s symptom.
  • Flaky / slow loop: a 30s flake is barely better than no loop; tighten until seconds and deterministic.
  • Product backlog drill: using live product issues for class practice instead of the Exercise Lab.

Diagnosis Loop — six phases (/diagnosing-bugs)

Order is mandatory. Phase 1 is the skill; everything else consumes the loop.

  1. Build a feedback loop — one red-capable command you have already run.
  2. Reproduce + minimise — confirm the user’s failure; shrink until every element is load-bearing.
  3. Hypothesise — 3–5 ranked, falsifiable predictions before testing any.
  4. Instrument — one variable at a time; probes map to predictions.
  5. Fix + regression test — failing test at a correct seam (when one exists), then fix; re-run original Phase 1.
  6. Cleanup + post-mortem — remove debug noise; state the winning hypothesis; ask what would have prevented this.

Phase 1 — Build a feedback loop (this is the skill)

Spend disproportionate effort here. Be aggressive. Be creative. Refuse to give up.

Phase 1 is done when you can name one command — a script path, test invocation, or curl — that you have already run at least once (paste invocation + output), and that is:

  • Red-capable — drives the actual bug path and asserts the reporter’s exact symptom (not merely “didn’t crash”).
  • Deterministic — same verdict every run (flakes: raise repro rate until debuggable).
  • Fast — seconds, not minutes.
  • Agent-runnable — unattended; HITL only via a structured harness if a human must click.

Ways to construct a loop (try roughly in order): failing test at a seam → curl/HTTP script → CLI + fixture → headless browser → trace replay → throwaway harness → fuzz → bisect harness → differential (old vs new) → HITL last.

Tighten: faster, sharper assertion, more deterministic. A 2-second deterministic loop is a debugging superpower.

If you cannot build a loop: stop. List attempts. Ask for env access, artifacts (HAR, logs, recording), or permission for temporary instrumentation. Do not proceed to hypothesise without a loop.

Phase 2 — Reproduce + minimise

  • Run the loop. Watch it go red — the bug appears.
  • Confirm it shows the user’s failure mode. Wrong bug = wrong fix.
  • Shrink inputs, callers, config, and steps one at a time; re-run after each cut.
  • Done when every remaining element is load-bearing — remove any one and the loop goes green.

A minimal repro shrinks the hypothesis space and becomes the clean regression test in Phase 5 / Module 4.

Phase 3 — Hypothesise

  • Generate 3–5 ranked hypotheses before testing any. Single-hypothesis thinking anchors on the first plausible idea.
  • Each must be falsifiable. Format: If X is the cause, then changing Y will make the bug disappear / changing Z will make it worse.
  • If you cannot state the prediction, the hypothesis is a vibe — discard or sharpen it.
  • Show the ranked list to your partner (or the reporter) when practical — cheap domain re-rank. Don’t block forever if they are AFK.

Phase 4 — Instrument

  • Each probe maps to a specific prediction from Phase 3. Change one variable at a time.
  • Prefer debugger / REPL → targeted logs → never “log everything and grep.”
  • Tag every debug log with a unique prefix (e.g. [DEBUG-a4f2]) so cleanup is a single grep.
  • Perf bugs: establish a baseline measurement first, then bisect — logs are usually wrong.

Phase 5 — Fix + regression test (preview — Module 4 owns the green path)

  • Prefer writing the regression test before the fix when a correct seam exists — one that exercises the real bug pattern at the call site.
  • If no correct seam exists, that itself is the finding: document it for architecture follow-up after the fix.
  • After the fix: re-run the original Phase 1 loop (un-minimised scenario).

In this module you only name the seam and the Phase 1 command. You do not ship the fix yet.

Phase 6 — Cleanup + post-mortem (ticket-level)

  • Original Phase 1 loop green; regression test passes (or seam absence documented).
  • All tagged debug instrumentation and throwaway harnesses removed or quarantined.
  • Winning hypothesis stated in the PR / commit message.
  • Ask: what would have prevented this? Park architectural deepening in a follow-up — after the fix is in.

Exercise Lab — Invoice API

Practice on the portable training fixture at exercises/invoice-api. It is not product code. Do not use a live product backlog for this drill.

Symptom: GET /v1/invoices?asOf=2026-07-16 returns 200 with an empty list when the client sends an Accept-Timezone header. Without that header, invoices return correctly. (Story: started after a fictional “timezone normalization” PR.)

Install once (from the repo):

cd exercises/invoice-api
npm install

Phase 1 red command (deterministic — no Cursor required):

npm run phase1

Expected while the bug is present: exit code 1; output includes RED — Phase 1 reproduce signal. The script starts an ephemeral server, hits the route with and without Accept-Timezone, then exits.

Optional manual curl (after npm start on port 3847) is in the lab README — useful for Phase 2 minimisation, but npm run phase1 is the canonical Phase 1 completion signal for this course.

Phase 1 completion criteria (lab)

You may check Phase 1 done for the Exercise Lab when all of the following are true:

  • You ran npm run phase1 from exercises/invoice-api at least once.
  • You captured exit code 1 and the RED — Phase 1 reproduce signal line (paste into notes / Evidence Pack later).
  • You can explain in one sentence which user symptom the script asserts (empty list with Accept-Timezone, non-empty without).
  • You have not started writing fix code yet.

Paired exercise (Exercise Lab — not product backlog)

With a partner (or an agent), work the Invoice API bug out loud. No fix code until your partner agrees Phases 1–3 are honest.

  1. Run npm run phase1. Paste the command and red output into a shared note.
  2. State a minimisation plan (what you would cut one-at-a-time: headers, query params, data, callers).
  3. Write 3–5 ranked falsifiable hypotheses for the empty-list-with-timezone symptom.
  4. Name the first instrumentation probe for hypothesis #1 (one variable).
  5. Propose the seam where the first regression test should live (public API vs list-logic boundary — agree which).

Stop here. Module 4 continues the same lab through red → green.

Sample ranked hypotheses (facilitator key)

One valid ranking for the Invoice API lab — not the only answer. Use after pairs draft their own list.

  1. Timezone normalization strips or mis-parses asOf when Accept-Timezone is present — if we log parsed filter date with/without header, empty list should correlate with a wrong cutoff.
  2. Header branch short-circuits to empty before query runs — if we instrument the list handler entry, we should see early return only when the header is set.
  3. Filter compares instants in wrong zone so all rows fall outside range — differential log of row timestamps vs filter bounds should show 100% exclusion with header.
  4. Fixture data keyed to “no header” path only — less likely if baseline without header returns three rows from same dataset.
  5. HTTP layer drops header before handler — probe at handler should still see Accept-Timezone if middleware is innocent.

Facilitator demo (optional)

  1. Narrate refusal to open product code until a red command exists.
  2. Show npm run phase1 going red on the lab.
  3. Minimise inputs live (header on/off, query params).
  4. Write 3–5 hypotheses on a whiteboard; invite re-ranking from the room.
  5. Stop before the fix — hand off to Module 4.

Checklist

Self-check

What comes first in the Diagnosis Loop — and what is Phase 1’s completion criterion?

Phase 1 — Build a feedback loop. Done when one command you have already run is red-capable (asserts the user’s exact symptom), deterministic, fast, and agent-runnable. For this course lab: npm run phase1 exiting 1 with RED — Phase 1 reproduce signal.

You have a plausible theory from reading invoices.ts. Should you test it now?

Not until Phase 1 is done. If you catch yourself theorising before a red-capable command exists, stop — that is the failure mode the skill prevents. Run (or build) the loop first; then hypothesise in ranked, falsifiable form.
