Why good bug processes matter

~15 min

Hard truth

A vague “it broke” ticket is not a bug report — it is a time sink. Without reproduction, severity, and a closed loop to prevention, the same class of bug returns.

Goals of this process

  • Reduce time-to-resolution
  • Improve reproducibility and root cause analysis
  • Enforce testing and prevent regressions
  • Build team muscle memory with Cursor skills
  • Create measurable improvement in bug quality and velocity

Core principles

  • Reproducibility first: if we cannot reliably reproduce it, we do not start fixing it.
  • Skills-driven: major steps use Matt Pocock skills (mattpocock/skills).
  • Documentation discipline: every ticket ends with root cause + prevention notes.
  • No shortcuts: TDD and code review are mandatory for production bugs.
  • Continuous improvement: review this SOP quarterly.

Common failure modes

  • Hero fixing: jump to a patch without a minimal reproduction.
  • Template skip: missing environment or impact — triage stalls.
  • Fix without test: same bug returns next sprint.
  • Close without prevention: no learning for the next person.

Do this now

Write one sentence: “The last bug that hurt us most was ___ because we lacked ___.” You will reuse this in Module 4.
