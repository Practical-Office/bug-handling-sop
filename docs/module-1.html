Bug Reporting Standards

~25 min

Learning objectives

What you will leave able to do

  • File a complete bug using the mandatory template without prompting.
  • Distinguish a weak report from an actionable one.
  • Use /grill-me or /grill-with-docs when the reporter’s story is incomplete.

Hard truth

Incomplete tickets get returned. Reporting is not optional polish — it is the first feedback loop. A vague “it broke” ticket is a time sink; do not guess missing fields.

Living SOP §4

Mandatory bug report template

All bug reports must use this template. Incomplete tickets are returned or moved to needs-info. Copy it:

**Title:** [Clear, concise description]

**Environment**
- Browser / OS / Version:
- Backend / API version:
- User role / permissions:
- Other relevant context (device, network, etc.):

**Reproduction Steps**
1.
2.
3.

**Expected Behavior**
[What should happen]

**Actual Behavior**
[What actually happens]

**Logs / Screenshots / Video**
[Attach or link here]

**Business / User Impact**
[Who is affected and how severely; note any workaround]

**Suggested Severity** (optional)
[severity/s1 … severity/s4 — triage confirms]

**Suggested Priority** (optional)
[priority/p0 … priority/p3 — triage confirms]

Teaching copy also lives at .github/ISSUE_TEMPLATE/bug-report.md.

Tips for great reports

  • Make reproduction steps as short and reliable as possible.
  • Include before / after states when relevant.
  • Prefer a screen recording with timestamps over a novel-length narrative.
  • Capture one failing API payload or console error when available.
  • Note workarounds (or “no workaround”) — they affect Severity in Module 2.
  • Title format that helps triage: [Area] verb + failure — e.g. [Checkout] Submit disabled after coupon apply.

Compare

Weak vs strong

Weak (do not accept)

“Dashboard is broken on mobile sometimes. Customers are complaining.”

Missing environment, STR, expected vs actual, evidence, and impact. Return it — do not invent details.

Strong (accept)

**Title:** [Dashboard] KPI cards blank on iOS Safari after refresh

**Environment**
- Browser / OS / Version: iOS 18.2, Safari; also reproducible in iPhone 15 Simulator
- Backend / API version: api v2.14.1 (staging + prod)
- User role / permissions: Org Admin
- Other: Wi-Fi only; does not reproduce on desktop Chrome 126

**Reproduction Steps**
1. Sign in as Org Admin on iPhone Safari.
2. Open Dashboard → Overview.
3. Pull to refresh (or hard-reload).
4. Observe KPI card row.

**Expected Behavior**
KPI cards show revenue, active users, and open tickets with numbers.

**Actual Behavior**
Cards render with labels but values stay “—” indefinitely. Network tab shows
`/api/metrics/summary` returning 200 with body `{ "error": "timezone_offset_invalid" }`
after refresh (first load is fine).

**Logs / Screenshots / Video**
- HAR attached; console screenshot of fetch error
- Loom: https://example.invalid/loom/kpi-blank

**Business / User Impact**
Org Admins cannot trust daily metrics on mobile (~30% of admin sessions).
Support tickets rising; not a full outage. Workaround: use desktop Chrome.

**Suggested Severity:** severity/s3
**Suggested Priority:** priority/p2

When intake is incomplete — grill

Do not start diagnosis or fix work on a half-empty template. Clarify first:

SituationSkillWhy
Reporter is a human with incomplete STR /grill-me Relentless Q&A until branches resolve
Ambiguity involves domain terms / product language /grill-with-docs Same grilling + updates CONTEXT.md / ADRs via domain modeling
Triage already running /triage (verify + grill if needed) Built-in path to grilling + routing (Module 2)

Workshop prompt: Facilitator plays a vague reporter. Engineer may only ask questions via a grill session — no codebase diving until the template is complete.

Paper Exercise A

Rewrite the weak report

Individually (~8 minutes): rewrite this into the mandatory template (note, draft issue, or paste below). Invent plausible but consistent details — mark anything you had to invent so a real triage would know to verify.

“Dashboard is broken on mobile sometimes. Customers are complaining.”

  1. Fill every required section (title through business/user impact).
  2. Optionally suggest severity/s* and priority/p* — triage will confirm.
  3. Peer review: mark any missing section; return incomplete tickets instead of guessing.

Optional pair drill: one person is reporter with a secret “real” bug; the other completes the template using /grill-me-style questions (even offline as a checklist).

A strong rewrite looks like the “Strong (accept)” example above: named area in the title, environment that narrows platform, numbered STR that hits the failure, expected vs actual with a concrete API error, evidence links, and impact with workaround noted. Your invented details will differ — the bar is completeness and falsifiability, not matching that story exactly.

Checklist

Self-check

A reporter says “checkout is broken.” Should you open /diagnosing-bugs immediately?

No. Complete the mandatory template first. Use /grill-me or /grill-with-docs until environment, STR, expected/actual, evidence, and impact are filled — then triage can verify the claim. Diagnosis starts only when Triage State is ready-for-agent or ready-for-human.
