Review, closure & prevention

~20–25 min

Code review & closure

  1. Open a PR with a clear description referencing the ticket.
  2. Run /code-review.
  3. Address all feedback.
  4. Update the original ticket with:
    • Root Cause: detailed explanation
    • Prevention: what we will do to stop this class of bug
  5. QA verifies the fix.
  6. Close ticket and PR.

Metrics to track

  • Average time from report → fix (by severity)
  • % of bugs with reproduction steps
  • % of fixes with regression tests
  • Recurring bug rate
  • Team feedback on process

Updating the living SOP

Suggested improvements or new skills should be proposed in a dedicated ticket on Practical-Office/bug-handling-sop. Approval: team lead + engineering manager. Edit BUG-HANDLING-SOP.md once approved, then align course modules.

Post-training

  • Each engineer completes 1–2 bugs following the full process.
  • Monthly retro on bug process effectiveness.
