Fixing with /tdd

~35 min

Learning objectives

  • Fix bugs with /tdd at pre-agreed seams — red → green in vertical slices.
  • Turn a minimised repro into a failing regression test that asserts the user’s symptom.
  • Keep the fix minimal and hypothesis-aligned; re-run the original Phase 1 loop after green.
  • Continue the Exercise Lab from Module 3 — not a product backlog ticket.

Hard truth

TDD without a failing test first is theater. Tests coupled to internals are liabilities. Speculative “while we’re here” refactors in a bug PR are how small fixes become large regressions.

Prerequisites from Module 3

Do not start this module’s lab path until you have:

  • A red Phase 1 signal from npm run phase1 in exercises/invoice-api.
  • A minimisation story and a ranked, falsifiable hypothesis (or short list).
  • A proposed correct seam for the regression test.

On a real ticket: Triage State must be ready-for-agent or ready-for-human before fix work starts. Diagnosis uses /diagnosing-bugs; the fix itself uses /tdd. Ticketed AFK work may use /implement, which drives /tdd then /code-review.

What a good regression test is

  • Verifies behavior through a public interface — not private helpers or implementation details.
  • Reads like a specification of the user’s symptom (“with Accept-Timezone, invoices for asOf date are returned”).
  • Survives refactors: it fails when the bug returns, not when you rename an internal.
  • Expected values come from an independent source of truth (known-good fixture / literal), not by recomputing the same way as the code.

Seams — where the test goes

A seam is the public boundary you observe without reaching inside. Tests live at seams.

  • Before writing any test, write down the seams under test and confirm with a human (partner / lead).
  • No test at an unconfirmed seam.
  • A correct seam exercises the real bug pattern at the call site. A shallow unit test that cannot reproduce the chain gives false confidence.
  • If no correct seam exists, document that finding — architecture may need a follow-up after the fix ships.

For the Invoice API lab, agree which boundary you will lock: HTTP route behavior (Phase 1 script / integration) and/or the list-logic function that mishandles timezone — as long as the assertion matches the reporter’s empty-list symptom.

Anti-patterns

Anti-patternTell
Implementation-coupled Breaks on refactor with unchanged behavior; mocks internals; tests private methods.
Tautological Assertion recomputes the expected value the same way the code does — can never disagree with the bug.
Horizontal slicing All tests first, then all implementation — you test imagined behavior, not tracer-bullet learning.

Rules of the loop (red → green → refactor)

  1. Red before green — write the failing test first; then only enough code to pass it.
  2. One slice at a time — one seam, one test, one minimal implementation per cycle.
  3. Refactoring is not part of the red→green loop — belongs with the /code-review stage (Module 5), not bundled into “make it green.”

Vertical slice mental model for bugs:

Slice 1: failing test for exact symptom at seam A → minimal fix
Slice 2: adjacent edge case diagnosis revealed → minimal fix
Stop when original Phase 1 loop is green and a Prevention note is draftable

Do not “while we’re here” rewrite the module. Park deepening in a follow-up ticket.

Recommended fix workflow

  1. Acknowledge the ticket; set work in progress (real bugs: after a ready-for-* Triage State).
  2. Finish /diagnosing-bugs through a clear root hypothesis (Module 3).
  3. Agree seams for regression coverage with a human.
  4. Run /tdd:
    • Turn the minimised repro into a failing test at the agreed seam.
    • Watch it fail for the right reason (the symptom — not a setup error).
    • Apply the minimal fix aligned with the winning hypothesis.
    • Watch green; then re-run the original Phase 1 command (un-minimised).
  5. Document findings on the ticket as you go (partial notes OK before PR). Keep the winning hypothesis visible in the PR description.

Closure, Root Cause + Prevention, and /code-review (Standards ‖ Spec) are Module 5 — do not skip them on real bugs; for the lab, draft a one-line Prevention note after green.

Common failure modes

  • Green without red: “fix” applied first; test written after to match the patch.
  • Wrong red: test fails on missing fixture / wrong assertion, not the user symptom.
  • Shotgun fix: remove the feature (e.g. drop Accept-Timezone globally) instead of honoring the contract.
  • Skip original loop: minimised case green, but the Phase 1 command still red (or never re-run).

Exercise D — Red → green (paired)

Continue the Module 3 Invoice API diagnosis. With your partner:

  1. Confirm the seam you named in Module 3 (write it down; partner must agree).
  2. Write a failing test sketch (real test in the lab, or pseudo-code if timeboxed) that asserts the user’s symptom: with Accept-Timezone set, invoices for the asOf date are not an empty list when data exists.
  3. Run or walk the sketch — it must fail for the right reason while the bug is present.
  4. Outline the minimal fix in ≤5 lines of prose (hypothesis-aligned). Then implement only that.
  5. Re-run npm run phase1. Expect green (exit 0).
  6. Self-review the test against the anti-pattern table before calling the slice done.

Pass criteria: the test would fail before the fix; the fix matches the winning hypothesis; the original Phase 1 command is green; no speculative refactors bundled in.

Facilitator demo (optional)

  3. Apply a minimal fix; watch the seam test green.
  4. Re-run Phase 1; confirm the un-minimised loop is green.
  5. Call out a bad fix (“remove Accept-Timezone support globally”) as a Spec fail for Module 5.

Checklist

Self-check

When do you write the regression test relative to the fix — and what must be true about the seam?

Write the failing regression test before the fix when a correct seam exists — one that exercises the real bug pattern at the call site. Confirm the seam with a human first. Watch red for the user’s symptom, then minimal fix, then green, then re-run the original Phase 1 command.

Your Phase 1 script is still red after the unit test is green. Are you done?

No. Re-run the original (un-minimised) Phase 1 loop. If it is still red, the seam test did not lock the real bug path — keep diagnosing or widen the seam until Phase 1 is green.
