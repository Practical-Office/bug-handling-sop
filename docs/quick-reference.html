Bug report template

Title · Environment · Reproduction Steps · Expected · Actual · Logs/Screenshots · Business/User Impact
(+ optional Suggested Severity / Priority — triage confirms)

Severity severity/s1–severity/s4

Impact (how bad). Workaround-aware. Not the same as Priority.

LabelMeaningTriage SLA
severity/s1Critical: outage, data loss, active security risk, core flow blocked — no acceptable workaround≤ 1h
severity/s2Major: significant break for many users — workaround complex/partial≤ 4h
severity/s3Moderate: limited/secondary — reasonable workaroundNormal cadence
severity/s4Low: cosmetic / polishNormal cadence

Priority priority/p0–priority/p3

Scheduling (when we work it), given Severity and capacity.

LabelMeaning
priority/p0Schedule now — interrupt
priority/p1Next — current sprint / immediate queue
priority/p2Soon — near-term backlog
priority/p3When capacity allows

Triage Category & State

Category (exactly one): bug · enhancement

State (exactly one):

StateMeaning
needs-triageMaintainer must evaluate
needs-infoWaiting on reporter (~14d stale → close/nudge)
ready-for-agentFully specified; agent brief attached
ready-for-humanNeeds human judgment / access / design
wontfixWill not action (with reason)

Work starts only at ready-for-agent or ready-for-human.

Diagnosis Loop — 6 phases

  1. Feedback loop — one red-capable, deterministic command
  2. Reproduce + minimise — shrink until every part is load-bearing
  3. Hypothesise — 3–5 ranked, falsifiable hypotheses
  4. Instrument — one variable at a time; tagged probes
  5. Fix + regression test — prefer test-before-fix at a correct seam; /tdd
  6. Cleanup + post-mortem — green loop, remove debug harnesses, ticket RCA
